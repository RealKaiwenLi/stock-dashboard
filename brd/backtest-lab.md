# 策略回测实验台 BRD

## 1. 背景与机会

背景：

- 项目已经有每日“纳斯达克指引”和每周“策略回测”脚本。
- 现有回测能力主要由 Python 配置和定时任务驱动，适合固定策略验证，但不适合用户临时尝试新组合。
- 用户希望在 dashboard 上快速选择股票、配置入场/退出规则、选择退出后的操作，并立即看到回测结果。

用户痛点：

- 每次想尝试一个新策略，都需要改 Python/config，反馈链路太长。
- 用户不仅想知道 QQQ/QLD 当前策略表现，也想快速测试其他组合，例如 NVDA/NVDL、SPY/SSO、TQQQ/QQQ、个股/现金。
- 普通回测工具通常参数很多、界面复杂，不适合个人策略假设的快速验证。
- 只看最终收益不够，用户需要同时理解最大回撤、波动、切换次数、最近信号和策略是否稳定。

机会判断：

- 一个面向个人使用的“策略实验台”可以把项目从静态报告升级成可交互研究工具。
- MVP 不需要做机构级策略平台，只要能覆盖“选择标的 + 配置规则 + 一键回测 + 看关键结果 + 保存候选策略”。

## 2. 产品定位

功能名称：

- 中文：策略回测实验台
- 英文可选：Backtest Lab

产品一句话：

- 用户可以在 dashboard 中用少量表单快速组合最多 5 个股票/ETF 轮动策略，并在数秒内并排比较历史表现、风险、交易记录和当前信号。

核心原则：

- 快速：用户 1 分钟内能配置并跑出一个基础回测，且可以继续添加候选策略做对比。
- 可解释：每个规则、指标和结果都要能看懂，不只给黑盒分数。
- 可比较：P0 支持最多 5 个自定义策略与基准 buy & hold 并排比较。
- 可保存：有价值的策略可以保存为候选方案，后续进入每日指引或每周回测。
- 不下单：页面不提供交易执行，只做研究和信号解释。

## 3. 目标用户与场景

目标用户：

- 主要用户是项目 owner 本人。
- 熟悉 ticker、ETF、现金、均线、MACD 等基础概念。
- 希望用中文理解策略逻辑，但 ticker 和技术术语可保留英文。

核心场景：

1. 用户想验证“QQQ 跌破某条均线后，QLD 是否应该退回 QQQ 或现金”。
2. 用户想测试“某个标的出现 MACD 金叉后切到杠杆 ETF，退出时换回原 ETF”的表现。
3. 用户想比较退出后换成现金、换成防守 ETF、换成原标的之间的差别。
4. 用户想快速改参数，例如 EMA10/15/20、MACD 8/17/9 vs 12/26/9。
5. 用户想保存表现较好的策略，后续让每日指引自动跟踪。
6. 用户想把多个策略变体放在同一张结果表里比较，例如 EMA10、EMA15、EMA20 或 Hist 条件开/关。

## 4. 问题定义

要解决的问题：

- 用 UI 表单配置常见轮动策略，而不是手改代码。
- 支持选择 signal asset、risk asset、fallback asset。
- 支持配置入场规则和退出规则。
- 支持选择退出后的操作：换回 signal asset、换现金、换到指定防守资产。
- 支持明确选择或展示执行方式，避免用户误解信号触发价和成交价。
- 支持多条件规则组合，至少能表达 `AND` / `OR`。
- 支持通用的退出后再入场策略：退出成交后可设置冷却期，并定义冷却期间入场信号是忽略还是暂存、暂存多久以及冷却结束后如何重新验证。
- 支持多标的数据对齐和数据质量摘要，避免不同标的历史长度不一致导致误读。
- 支持运行回测并展示关键绩效和交易明细。
- 支持一次添加并比较最多 5 个策略候选。
- 支持把常用策略保存为“策略收藏”，并从收藏中一键添加到策略候选。

不解决的问题：

- 不做券商下单。
- 不做分钟级高频回测。
- 不做复杂 portfolio optimizer。
- 不做多资产权重再平衡策略。
- 不提供脚本编辑器、表达式语言或任意代码执行；退出后再入场策略只能通过受支持的表单字段和规则目录配置。
- 不做期权或保证金。
- P0/P1 不做手续费、滑点、bid/ask spread 或税务建模；成本模型先不进入范围。
- P0/P1 不做资产可交易性、流动性、杠杆 ETF 衰减或相关性审查。
- P0/P1 不做样本内/样本外、walk-forward 或参数稳定性验证。
- P0/P1 不做策略生命周期管理，例如 Draft/Candidate/Active/Retired。
- MVP 不做用户账户和云端多用户协作。

关键假设：

- MVP 使用日线数据即可支撑当前策略研究。
- 用户接受 P0 中每个策略候选都使用同一类策略模型：在 signal/risk/fallback 三类资产之间切换。
- 用户接受 P0 最多同时比较 5 个策略，超过 5 个需要删除或替换已有候选。
- 回测结果用于研究，不等于未来收益预测。

## 5. MVP 范围

P0 必须有：

- 新页面入口：`Backtest Lab` / `策略回测`。
- 策略配置表单。
- 数据范围配置。
- 执行方式展示，默认 `T 日收盘确认 / T+1 开盘成交`。
- 数据对齐摘要。
- 入场规则配置。
- 退出规则配置。
- 入场/退出多条件组合，至少支持 `AND` / `OR`。
- 退出后操作配置。
- 退出后再入场策略配置；默认关闭，关闭时必须完全保持现有信号和成交行为。
- 策略候选列表，最多 5 个。
- 一键运行回测。
- 多策略结果摘要和排名表。
- 与基准对比。
- 多策略权益曲线。
- 单策略交易记录表。
- 多策略当前信号解释。
- 收藏策略模板到“策略收藏”（MVP 使用浏览器存储）。
- 从策略收藏一键添加策略候选。
- 明确免责声明。

P1 应该有：

- 参数网格搜索。
- 策略排名和评分。
- 批量生成策略变体。
- 退出后再入场策略的更多信号队列方式、按退出原因配置不同策略，以及事件统计的策略间比较。
- CAPE 风险过滤：基础入场/退出规则维护趋势状态，CAPE 决定是否允许持有 Risk Asset。
- 保存策略到每日指引配置。
- 从每周回测结果一键加载策略。
- 回测结果导出 Markdown/JSON。

P2 可考虑：

- 多资产组合权重。
- 再平衡策略。
- 移动端原生通知联动。

## 6. 页面信息架构

页面分为 5 个区域：

1. 顶部策略栏
2. 策略配置区
3. 运行与结果摘要区
4. 图表区
5. 明细与保存区

### 6.1 顶部策略栏

显示：

- 页面标题：策略回测实验台
- 当前实验名称输入框
- 数据状态
- 最近一次运行时间
- 保存按钮
- 重置按钮

示例：

```text
实验名称：QQQ/QLD EMA 退出参数对比
数据范围：2014-01-01 至 2026-06-26
状态：已运行
```

### 6.2 策略配置区

配置区采用分组表单，不做代码编辑器。

P0 支持策略候选列表：

- 页面至少有 1 个策略候选。
- 用户最多可以添加 5 个策略候选。
- 每个策略候选都有独立名称、标的设置、入场规则、退出规则和退出后操作。
- 每个策略候选可以独立配置退出后再入场策略；该配置属于候选策略本身，不是实验级共享设置。
- 数据范围、执行设置和 benchmark 在 P0 中对所有策略候选共享。
- 用户可以复制一个策略候选并修改参数，例如复制 EMA15 策略后改成 EMA20。
- 用户可以收藏一个策略候选；收藏保存该策略的配置模板，不保存回测结果。
- 用户可以从本地收藏一键添加一个策略候选；添加后生成新的候选策略，可继续编辑。
- 当已有 5 个策略候选时，`添加策略` 按钮 disabled，并提示“最多同时比较 5 个策略”。
- 用户可以删除非最后一个策略候选；页面至少保留 1 个策略。

策略候选列表展示字段：

| 字段 | 说明 |
| --- | --- |
| 策略名称 | 用户可编辑，例如 EMA15 + Hist |
| Signal / Risk / Fallback | 标的组合 |
| 入场摘要 | 自然语言规则摘要 |
| 退出摘要 | 自然语言规则摘要 |
| 当前是否已运行 | 是否已有最新回测结果 |

#### A. 标的设置

字段：

| 字段 | 类型 | 示例 | 说明 |
| --- | --- | --- | --- |
| Signal Asset | ticker search | QQQ | 用于计算信号的基础资产 |
| Risk Asset | ticker search | QLD | 入场后持有的风险资产 |
| Fallback Asset | ticker search / CASH | QQQ / CASH / SHY | 退出后切换到哪里 |
| Benchmark | ticker search | QQQ | 对比基准 |

默认值：

- Signal Asset: `QQQ`
- Risk Asset: `QLD`
- Fallback Asset: `QQQ`
- Benchmark: `QQQ`

验收标准：

- 用户可以输入 ticker 并选择标的。
- `Fallback Asset` 必须支持 `CASH`。
- 如果 Risk Asset 与 Signal Asset 相同，页面提示“该策略不会产生风险资产切换”。

#### B. 数据设置

字段：

| 字段 | 类型 | 默认 | 说明 |
| --- | --- | --- | --- |
| Start Date | date | 10 年前 | 回测开始日期 |
| End Date | date | 最新交易日 | 回测结束日期 |
| Bar Interval | select | 1D | MVP 仅支持日线 |
| Initial Capital | number | 10000 | 初始资金，仅用于展示 |

P0 限制：

- 只支持 `1D` 日线。
- 如果数据不足，显示“数据不足，无法计算完整指标”。

数据对齐要求：

- 页面必须显示每个参与标的的数据开始日期、结束日期和有效行数。
- 回测必须使用所有参与标的共同拥有数据的重叠区间。
- 页面必须显示“对齐后有效回测区间”。
- 如果 signal、risk、fallback 或 benchmark 的交易日不完全一致，系统必须按日期 inner join 对齐，并显示被丢弃的日期数量。
- 如果对齐后数据不足以计算指标，用户不能运行回测。

示例数据摘要：

| 项目 | 数值 |
| --- | --- |
| QQQ 数据区间 | 1999-03-10 至 2026-06-26 |
| QLD 数据区间 | 2006-06-21 至 2026-06-26 |
| 对齐后回测区间 | 2006-06-21 至 2026-06-26 |
| 对齐后有效日线 | 5032 |

#### C. 执行设置

P0 默认执行方式：

```text
T 日收盘确认信号 / T+1 开盘成交
```

字段：

| 字段 | 类型 | 默认 | 说明 |
| --- | --- | --- | --- |
| 信号确认时点 | readonly | T 日收盘 | 使用收盘价确认信号 |
| 成交时点 | readonly | T+1 开盘 | 下一交易日开盘切换持仓 |
| 数据缺口处理 | readonly | 下一可用交易日 | 如果 T+1 无数据，使用下一条有效日线 |

验收标准：

- 页面必须显示执行方式，不能只隐藏在说明文案里。
- 交易记录必须同时显示 `Signal Date` 和 `Execution Date`。
- 如果信号日后一交易日缺失，系统必须使用下一可用交易日，并在交易记录中标注。

#### D. 入场规则

MVP 支持规则类型：

1. MACD 金叉
2. 收盘价站上均线
3. 价格突破 N 日高点

P0 页面可以预填一个示例策略，但不能把 MACD 金叉作为唯一策略模型；用户必须能自己选择一个或多个入场规则并调节对应参数。

```text
当 Signal Asset 的 MACD 线上穿 Signal 线时，下一交易日开盘切到 Risk Asset。
```

表单字段：

| 字段 | 类型 | 默认 | 说明 |
| --- | --- | --- | --- |
| 入场规则组合方式 | segmented | AND | 多个入场规则全部满足或任一满足 |
| 入场规则列表 | repeatable rule rows | MACD 金叉 | 用户可以添加、删除、修改多个规则；至少保留 1 条 |
| 入场规则类型 | select | 示例可为 MACD 金叉 | 每条规则可选 MACD 金叉 / 收盘价站上均线 / 突破 N 日高点 |
| MACD Fast | number | 12 | 仅 MACD 金叉显示；快线 EMA 周期 |
| MACD Slow | number | 26 | 仅 MACD 金叉显示；慢线 EMA 周期 |
| MACD Signal | number | 9 | 仅 MACD 金叉显示；Signal 线周期 |
| 均线类型 | select | EMA | 仅站上均线显示；EMA / SMA |
| 均线周期 | number | 50 | 仅站上均线显示 |
| 突破周期 | number | 20 | 仅突破 N 日高点显示 |

验收标准：

- 用户可以选择入场规则类型；不同规则类型只显示对应参数。
- 用户可以修改所选规则的参数，不能被固定在 MACD。
- 页面解释 `MACD`、`Signal`、`Hist` 的含义。
- 入场规则必须显示自然语言摘要。

#### F. 风险过滤

P1 支持独立于入场/退出规则组的 CAPE 风险过滤：

- 用户可以为每个策略单独启用或关闭 CAPE 过滤。
- 用户可以配置允许持有 Risk Asset 的 CAPE 上限，默认 `30`。
- 未启用时，策略行为与当前入场/退出规则完全一致。
- 启用后，基础入场规则触发时进入 `Risk On` 状态；仅当可获得的 CAPE 小于等于阈值时持有 Risk Asset，否则持有 Fallback Asset。
- 若基础趋势仍为 `Risk On`，CAPE 从高于阈值恢复到小于等于阈值时，不需要等待新的入场交叉信号，下一交易日重新进入 Risk Asset。
- 基础退出规则触发后离开 `Risk On` 状态，即使 CAPE 满足阈值也继续持有 Fallback Asset，直到下一次基础入场信号。
- CAPE 使用月度数据，并至少延迟到下一个月才参与信号计算，避免未来数据泄漏。
- 当前信号详情必须展示最新可用 CAPE、阈值及是否通过。
- CAPE 数据获取失败时，本次回测明确失败，不得静默忽略过滤条件。
- P0 入场条件支持多个规则组合。
- 用户可以为入场规则添加多条条件，并可删除非最后一条条件。
- 当入场规则有多条条件时，用户必须选择 `全部满足 AND` 或 `任一满足 OR`；只有一条规则时仍展示该设置以保持一致。
- `Hist > 0` 不作为特殊 checkbox 出现在入场规则中；如未来要支持 MACD Hist 入场过滤，应作为普通规则类型添加。

示例摘要：

```text
入场：当 QQQ 的 MACD(12,26,9) 线上穿 Signal 线时，下一交易日开盘买入 QLD。
```

#### E. 退出规则

MVP 支持规则类型：

1. 收盘价跌破均线
2. MACD 死叉
3. 价格跌破 N 日低点
4. MACD Hist > 0

P0 页面可以预填一个示例退出规则组，但不能把 EMA、SMA 或 `Hist > 0` 特殊化为独立 checkbox；用户必须能自己选择一个或多个退出规则并调节对应参数。

```text
当 Signal Asset 收盘价跌破 EMA15，且 Hist > 0 时，下一交易日开盘退出 Risk Asset。
```

表单字段：

| 字段 | 类型 | 默认 | 说明 |
| --- | --- | --- | --- |
| 退出规则组合方式 | segmented | AND | 多个退出规则全部满足或任一满足 |
| 退出规则列表 | repeatable rule rows | 跌破 EMA15 + Hist > 0 | 用户可以添加、删除、修改多个规则；至少保留 1 条 |
| 退出规则类型 | select | 示例可为跌破均线 | 每条规则可选跌破均线 / MACD 死叉 / 跌破 N 日低点 / Hist > 0 |
| 均线类型 | segmented | EMA | 仅跌破均线显示；EMA / SMA |
| 均线周期 | number | 15 | 仅跌破均线显示，例如 10/15/20/50/200 |
| MACD Fast/Slow/Signal | number | 12/26/9 | 仅 MACD 死叉显示 |
| 跌破周期 | number | 20 | 仅跌破 N 日低点显示 |
| Hist MACD Fast/Slow/Signal | number | 12/26/9 | 仅 Hist > 0 显示；用于计算 MACD Hist |
| 是否启用 SMA200 防守 | toggle | off | 跌破 SMA200 是否转现金 |

验收标准：

- 用户可以把 `Hist > 0` 作为普通退出规则添加、删除或改参数，而不是只能操作一个退出附加条件 checkbox。
- 用户可以选择退出规则类型；不同规则类型只显示对应参数，不能被固定在均线。
- 页面必须把退出条件拆成 checklist，不能只显示最终结论。
- 用户可以为退出规则添加多条条件，并可删除非最后一条条件。
- 当退出规则有多条条件时，用户必须选择组合逻辑是 `AND` 还是 `OR`；只有一条规则时仍展示该设置以保持一致。
- 回测结果必须展示每个退出子条件的最新状态。
- 交易记录中的 `Reason` 必须写明触发的是哪一组条件。

示例摘要：

```text
退出：当 QQQ 收盘价低于 EMA15，且 Hist > 0 时，下一交易日开盘从 QLD 切回 QQQ。
```

#### F. 退出后操作

用户可以选择退出后怎么操作。

选项：

| 选项 | 行为 | 示例 |
| --- | --- | --- |
| 换回 Signal Asset | 退出 Risk 后买回 signal | QLD -> QQQ |
| 换到现金 | 退出 Risk 后持有现金 | QLD -> CASH |
| 换到指定防守资产 | 退出 Risk 后买入 fallback | QLD -> SHY |

验收标准：

- 用户必须明确选择退出后持有什么。
- 如果选择现金，回测应展示现金收益假设。
- 如果选择防守资产，必须要求输入 fallback ticker。

#### G. 退出后再入场策略（Post-exit Re-entry Policy）

产品目标：

- 允许用户以通用表单表达“退出 Risk Asset 后等待若干交易日，再决定是否重新进入”的策略。
- 该能力只依赖抽象的退出事件、入场规则、交易日和资产角色，不绑定任何特定 ticker、指标或参数。
- 页面不得出现针对某个研究模型硬编码的开关或固定数值；所有示例参数都只是可编辑示例。

P0 产品模型：

```text
Risk Asset 退出成交
  -> 可选冷却期
  -> 冷却期间按策略忽略或暂存入场信号
  -> 暂存信号到期、被更新，或等待冷却结束
  -> 冷却结束后按释放验证规则判断
  -> 验证通过后生成再入场指令
  -> 下一可用交易日开盘成交
```

配置字段：

| 字段 | 类型 | 默认 | 说明 |
| --- | --- | --- | --- |
| 启用退出后再入场策略 | toggle | off | 关闭时完全沿用现有行为，不创建冷却或待处理信号 |
| 冷却交易日数 | integer | 10 | 从实际退出成交日开始建立冷却；成交日记为第 0 日，之后 N 个对齐后的有效交易日禁止再入场；第 N 日收盘完成释放判断，通过后最早在第 N+1 个有效交易日开盘成交 |
| 冷却期间入场信号 | select | 忽略 | `忽略` / `暂存最新信号` |
| 信号保留交易日数 | integer | 5 | 仅“暂存最新信号”时显示；信号日为有效期第 1 日，有效期包含信号日及之后 N-1 个有效交易日 |
| 冷却结束后的释放验证 | select | 重新验证入场规则 | `暂存信号仍有效即可` / `重新验证入场规则` / `通过释放验证规则组` |
| 释放验证规则组合方式 | segmented | AND | 仅选择“通过释放验证规则组”时显示；支持 AND / OR |
| 释放验证规则列表 | repeatable rule rows | 无 | 使用受支持的规则目录与参数表单，不允许输入脚本或任意表达式 |

释放验证规则目录要求：

- 规则构建器必须保持资产和指标通用，由用户选择参与判断的资产角色、规则类型和参数。
- P0 至少允许复用现有规则目录，并提供事件规则对应的“当前状态”形式，例如“快线当前高于慢线”“收盘价当前高于/低于均线”。
- “重新验证入场规则”表示在释放判断日收盘重新计算完整入场规则组；如果入场规则包含“上穿”“突破”等事件型条件，则只有该释放判断日本身再次发生该事件才算满足。
- 如果用户希望保留一次事件型信号，但在释放时检查趋势是否仍有效，应选择“通过释放验证规则组”，并配置相应的当前状态规则，而不是修改原始入场规则的含义。

状态与时间语义：

- 冷却只能由 `Risk Asset -> Fallback Asset/CASH` 的实际成交启动；退出信号日不启动冷却，未成交或被数据错误阻断的退出也不启动冷却。
- 退出成交日为冷却第 0 日，不计入 N 个完整冷却交易日。
- “交易日”使用该策略完成数据对齐后的有效日线，不使用自然日；周末、市场休市日和因 inner join 被丢弃的日期均不计数。
- N 个冷却交易日内不得执行进入 Risk Asset 的交易。
- 冷却结束是状态变化，不是成交事件。系统在第 N 个冷却交易日收盘后解除限制，并在该时点检查尚未到期的暂存信号；验证通过时，最早于下一可用交易日开盘成交。
- 第 N 个冷却交易日收盘的处理顺序为：先完成冷却计数并解除限制，再评估该日新产生的入场信号；因此该日收盘的新信号视为冷却结束后的正常信号，可直接生成 T+1 指令，不进入暂存槽。
- 在第 N 个冷却交易日收盘，系统先处理此前暂存信号的到期与释放，再评估该日新信号。若两者都指向同一次 Risk Asset 再入场，只生成一个 T+1 指令，并在事件原因中同时关联暂存信号与当日新信号。
- 当冷却期间入场信号设为“忽略”时，信号不创建待处理状态；冷却结束后必须等待新的正常入场信号。
- 当设为“暂存最新信号”时，策略最多保留一个待处理入场信号。新的入场信号替换旧信号，并以新信号日重新计算有效期；交易事件明细必须记录旧信号被替换。
- 同一交易日内由一个入场规则组产生的多个满足条件只合并为一个入场信号，不重复创建或延长有效期。
- 暂存信号在有效期最后一个交易日收盘前仍有效；超过有效期且尚未释放时标记为 `Expired`，之后冷却结束也不得执行。
- 如果信号到期与冷却结束发生在同一交易日，先按该日收盘数据完成释放验证，再在验证失败或未执行时标记到期。
- 释放验证失败后，该暂存信号立即标记为 `Rejected`，不会自动继续等待；后续再入场需要新的入场信号。
- 释放验证通过只生成待执行指令；实际成交仍遵循统一的 `T 日收盘确认 / T+1 开盘成交`，如果下一交易日缺数据则顺延到下一可用交易日。
- 生成待执行指令后、成交前若出现新的退出条件，不取消该指令；P0 只在持有 Risk Asset 时评估退出规则。该限制必须在口径说明中展示。

Disabled 兼容性：

- `启用退出后再入场策略 = off` 时，系统不得启动冷却、暂存信号或执行释放验证。
- 关闭状态下，策略的交易序列、绩效指标、当前持仓和当前信号必须与引入该能力之前、相同其余配置的回测结果完全一致。
- 保存旧版配置时缺少该字段，加载后必须按 `off` 解释，不允许自动套用默认冷却参数。
- 关闭开关时，其他再入场字段可以保留在草稿中但不参与回测；自然语言摘要应显示“退出后再入场限制：关闭”，不得暗示参数已生效。

配置区交互：

- “退出后再入场策略”作为每个候选策略内的独立折叠区，放在“退出后操作”之后。
- 开关关闭时隐藏或 disabled 其余字段，同时展示“关闭后，新的入场信号按原策略立即生成 T+1 指令”。
- 选择“忽略”时隐藏“信号保留交易日数”和全部释放验证字段；这些隐藏字段即使保留在草稿中也不参与回测。
- 选择“暂存最新信号”时显示保留期和释放验证方式。
- 选择“通过释放验证规则组”时展示规则组合方式和规则列表；选择其他验证方式时不展示该规则组。
- 表单旁必须展示基于当前配置生成的自然语言摘要，并随着字段修改实时更新。

通用摘要示例：

```text
退出成交后冷却 8 个交易日。冷却期间暂存最新入场信号，信号保留 4 个交易日。
冷却结束时，若信号仍有效且释放验证规则全部满足，则生成再入场指令，并于下一可用交易日开盘执行。
```

该示例中的资产、规则类型和数值不是产品约束。

## 7. 回测运行行为

用户点击 `运行回测` 后：

1. 校验表单。
2. 拉取或读取历史数据。
3. 计算指标。
4. 对每个策略候选生成交易信号。
5. 对已启用退出后再入场策略的候选，更新冷却计数与待处理信号生命周期，并执行到期或释放验证。
6. 对每个策略候选按 T 日收盘确认 / T+1 开盘成交规则模拟持仓。
7. 对每个策略候选计算绩效指标。
8. 生成 benchmark 结果。
9. 展示多策略比较结果。

P0 运行要求：

- 一次运行最多处理 5 个策略候选。
- 如果某个策略配置非法，只阻塞该策略，并在候选列表与结果表中显示错误。
- 如果共享数据设置非法，例如日期区间无效或 benchmark 无数据，则阻塞整次运行。
- 所有策略候选必须使用相同数据范围、执行方式和 benchmark，保证比较公平。
- 同一个策略的冷却计数、信号有效期和释放验证必须使用同一套对齐后有效交易日序列。
- 重新运行时必须从初始状态重新计算冷却与待处理信号，不能沿用上一次运行的运行时状态。

运行状态：

- Idle
- Validating
- Fetching Data
- Running Backtest
- Complete
- Error

错误场景：

- ticker 不存在。
- 数据不足。
- 规则参数非法。
- 回测结果为空。
- 数据源请求失败。

错误文案示例：

```text
QQQ 数据不足：当前只有 120 条日线，MACD(12,26,9) + EMA200 至少需要 220 条以上日线。
```

## 8. 结果展示

### 8.1 结果摘要卡片

必须展示多策略排名表。每行代表一个策略候选，外加至少一个 benchmark 行。

| 指标 | 说明 |
| --- | --- |
| Rank | 策略排名 |
| Strategy | 策略名称 |
| Total Return | 总收益 |
| CAGR | 年化收益 |
| Max Drawdown | 最大回撤 |
| Sharpe | 风险调整收益 |
| Annual Volatility | 年化波动 |
| Switches | 切换次数 |
| Win vs Benchmark | 是否跑赢基准 |
| Latest Signal | 最新信号 |
| Current Holding | 当前持仓 |
| Deferred Entries | 冷却期间被暂存的入场信号数 |
| Expired / Rejected | 到期或释放验证失败的暂存信号数 |

排序要求：

- 默认按 CAGR 降序排序。
- 用户可以按 CAGR、Max Drawdown、Sharpe、Switches 排序。
- Benchmark 行不参与策略排名，但必须固定显示。
- 如果某个策略运行失败，该行显示错误状态，不阻塞其他策略结果展示。

### 8.2 多策略 vs 基准

展示：

- 最多 5 条策略权益曲线。
- Benchmark 权益曲线。
- Risk Asset buy & hold 曲线可选展示。
- 回撤曲线 P1。

验收标准：

- 用户能看到策略是否只是因为杠杆而收益更高。
- 如果策略最大回撤高于基准，应有风险提示。
- 用户可以通过图例开关隐藏/显示某个策略曲线。
- 曲线颜色必须稳定绑定策略候选，避免重跑后颜色变化导致误读。
- 图表必须显示 x 轴日期刻度与 y 轴权益数值刻度。
- 鼠标 hover 曲线区域时，页面必须显示对应日期和各可见曲线在该日期的权益数值。

### 8.3 当前信号解释

必须支持查看每个策略的最新交易日判断。

| 条件 | 数值 | 是否满足 |
| --- | --- | --- |
| QQQ 收盘价 < EMA15 | 722 < 724 | 是 |
| Hist > 0 | -3.1 | 否 |
| 完整退出条件 | - | 否 |

并显示自然语言解释：

```text
当前持有 QLD。虽然 QQQ 已跌破 EMA15，但 Hist 不大于 0，因此退出条件未完整触发，下一交易日继续持有 QLD。
```

当启用退出后再入场策略时，还必须显示：

| 项目 | 说明 |
| --- | --- |
| 再入场状态 | `Inactive` / `Cooling Down` / `Pending Signal` / `Ready for Validation` / `Order Pending` |
| 冷却起点 | 实际退出成交日 |
| 冷却进度 | 已完成交易日数 / 配置交易日数 |
| 最早释放判断日 | 按对齐交易日推算的日期；数据未覆盖时显示“超出当前数据范围” |
| 待处理信号 | 信号日、来源规则摘要和状态 |
| 信号有效期 | 最后有效交易日和剩余有效交易日数 |
| 释放验证 | 验证模式、各子条件的当前值与是否满足 |
| 下一步动作 | 等待、忽略、到期、重新验证、已生成 T+1 指令或继续正常规则 |

自然语言解释示例：

```text
当前持有 Fallback Asset，处于退出后的冷却期，已完成 6/8 个交易日。
最近一次入场信号已暂存，将在 2 个交易日后到期；若冷却结束时仍有效，系统会按配置重新验证后决定是否生成下一交易日开盘指令。
```

P0 展示方式：

- 默认展示排名第一策略的当前信号解释。
- 用户可以在策略 tabs 或 selector 中切换查看其他策略的当前信号解释。
- 多策略排名表中必须直接显示每个策略的 `Current Holding` 和 `Latest Signal`。
- 开关关闭时不展示伪造的冷却状态，并明确显示“退出后再入场限制未启用”。
- 如果当前存在暂存信号或待执行指令，解释区必须区分“信号已出现”“允许释放”“已生成指令”和“已经成交”，不能把未成交状态描述为当前持仓。

### 8.4 交易记录

表格字段：

| 字段 | 说明 |
| --- | --- |
| Signal Date | 触发信号的日期 |
| Execution Date | 执行切换的日期 |
| From | 原持仓 |
| To | 新持仓 |
| Reason | 触发原因 |
| Execution Price | 执行价格 |
| Equity After Trade | 切换后权益 |

验收标准：

- 用户可以选择一个策略查看交易记录。
- 用户能看到该策略每次切换是为什么发生的。
- 交易记录可以按日期倒序查看。
- P0 不要求同时展示多个策略的交易记录。
- 因冷却导致的暂存、替换、到期或拒绝不是实际交易，不能伪装成交易记录。

### 8.5 策略事件明细

P0 为每个策略提供可筛选的事件明细，用于解释交易之外的信号生命周期。

字段：

| 字段 | 说明 |
| --- | --- |
| Event Date | 事件发生的有效交易日 |
| Event Type | `Exit Executed` / `Cooldown Started` / `Entry Ignored` / `Signal Retained` / `Signal Replaced` / `Signal Expired` / `Release Passed` / `Release Rejected` / `Order Scheduled` |
| Holding | 事件发生后实际持仓 |
| Cooldown Progress | 当前冷却计数；不适用时为空 |
| Signal Date | 关联的原始入场信号日；不适用时为空 |
| Valid Through | 暂存信号最后有效交易日；不适用时为空 |
| Rule Snapshot | 触发规则或释放验证规则的值与判断结果 |
| Explanation | 面向用户的自然语言原因 |

验收标准：

- 用户可以在“交易记录”和“策略事件”之间切换。
- 默认按日期倒序显示，并可按 Event Type 筛选。
- 每次实际退出成交、冷却开始、信号忽略/暂存/替换/到期、释放验证和 T+1 指令生成都可追溯。
- 事件日期、信号日期和执行日期必须使用不同字段，不得混为一个“交易日期”。
- 同一运行使用确定性的事件顺序；同日事件按“成交 -> 冷却状态更新 -> 既有暂存信号到期/释放判断 -> 当日新信号判断 -> 指令生成”展示。

## 9. 保存策略

用户可以保存整个回测实验，也可以保存单个策略候选为“策略草稿”。

保存字段：

- 实验名称
- 共享数据范围
- 共享执行设置
- Benchmark
- 策略候选列表，最多 5 个
- 每个策略候选的名称
- 每个策略候选的 Signal Asset
- 每个策略候选的 Risk Asset
- 每个策略候选的 Fallback Asset
- 每个策略候选的入场规则
- 每个策略候选的退出规则
- 每个策略候选的退出后再入场策略开关、冷却交易日数、冷却期间信号处理、保留交易日数、释放验证方式和释放验证规则组
- 每个策略候选的最新回测摘要
- 创建时间
- 更新时间

保存行为：

- `保存实验` 保存当前最多 5 个策略候选和共享设置。
- `保存当前策略` 只保存当前选中的单个策略候选。
- 重新加载实验后，候选策略数量、名称、颜色和排序必须保持一致。
- 重新加载实验或策略收藏后，退出后再入场配置必须逐字段恢复，并生成与保存前一致的自然语言摘要。
- 保存只持久化配置与可选的最新结果摘要，不持久化 `Cooling Down`、待处理信号或待执行指令等运行时状态；加载后必须重新运行回测才能恢复结果状态。
- 旧版本保存对象缺少退出后再入场字段时，必须以 `enabled = false` 向后兼容。

P0 存储：

- 可以先保存在浏览器 `localStorage`。

P1 存储：

- 保存到后端或 Notion database。
- 可标记为“用于每日指引”。

## 10. 用户故事与验收标准

### US1：用户可以配置基础 QQQ/QLD 策略

作为用户，我希望选择 QQQ 作为 signal、QLD 作为 risk、QQQ 作为 fallback，以便测试当前纳斯达克策略。

验收标准：

- Given 用户打开策略回测实验台
- When 页面加载默认配置
- Then Signal Asset 为 QQQ
- And Risk Asset 为 QLD
- And Fallback Asset 为 QQQ
- And 入场规则为 MACD(12,26,9) 金叉
- And 退出规则为跌破 EMA15 且 Hist > 0

### US2：用户可以修改退出后操作

作为用户，我希望退出 QLD 后可以选择换回 QQQ、现金或防守资产，以便比较不同防守方式。

验收标准：

- Given 用户正在配置策略
- When 用户选择“换到现金”
- Then Fallback Asset 显示为 CASH
- And 回测结果展示现金收益假设
- When 用户选择“换到指定防守资产”
- Then 页面要求输入 fallback ticker

### US3：用户可以关闭 Hist > 0 条件

作为用户，我希望能删除或替换 Hist > 0 退出规则，以便测试“跌破 EMA 就退出”或其他规则组合的表现。

验收标准：

- Given 退出规则组包含“收盘价跌破 EMA15”和“Hist > 0”
- When 用户删除“Hist > 0”规则
- Then 策略摘要更新为“跌破 EMA15 即退出”
- And 回测结果基于新规则重新计算

### US3A：用户可以为入场和退出配置多条规则

作为用户，我希望入场规则和退出规则都能多选，并选择 AND 或 OR，以便表达更复杂但仍可解释的策略。

验收标准：

- Given 用户正在编辑策略候选
- When 用户在入场规则组中添加第二条规则
- Then 页面展示第二条入场规则的类型选择和对应参数
- And 用户可以选择入场规则组合方式为 AND 或 OR
- When 用户在退出规则组中添加 `Hist > 0`
- Then `Hist > 0` 出现在退出规则列表中，而不是独立的“退出附加条件” checkbox
- And 用户可以选择退出规则组合方式为 AND 或 OR
- And 用户不能删除规则组中的最后一条规则

### US4：用户可以运行回测并看到关键结果

作为用户，我希望点击运行后看到最多 5 个策略的收益、回撤、Sharpe 和切换次数，以便判断哪个策略值得继续研究。

验收标准：

- Given 用户配置合法策略
- When 点击运行回测
- Then 页面展示 loading 状态
- And 回测完成后展示多策略结果摘要和排名表
- And 展示最多 5 条策略权益曲线与 benchmark 权益曲线
- And 用户可以选择单个策略查看交易记录表

### US5：用户可以查看当前信号解释

作为用户，我希望回测结果告诉我每个策略最新交易日应该持有什么，以及为什么。

验收标准：

- Given 回测已完成
- Then 页面展示默认策略的当前持仓
- And 展示次日开盘动作
- And 展示条件 checklist
- And 展示自然语言解释
- When 用户切换到另一个策略
- Then 当前信号解释同步更新为该策略的判断

### US6：用户可以保存策略草稿

作为用户，我希望保存当前实验配置和策略候选，以便下次继续调整或加入每日指引。

验收标准：

- Given 用户已配置并运行策略
- When 点击保存实验
- Then 当前实验的共享设置和最多 5 个策略候选保存到本地
- And 页面显示保存成功
- And 用户刷新页面后可以加载保存的实验

### US7：用户可以添加最多 5 个策略候选并比较

作为用户，我希望把多个策略变体放在同一次回测里比较，以便快速判断参数差异。

验收标准：

- Given 用户打开策略回测实验台
- When 用户点击“添加策略”
- Then 页面新增一个策略候选
- And 用户可以编辑该策略候选的名称、标的、入场规则、退出规则和退出后操作
- When 页面已有 5 个策略候选
- Then “添加策略”按钮不可用
- And 页面提示“最多同时比较 5 个策略”
- When 用户运行回测
- Then 结果表展示所有有效策略候选和 benchmark

### US8：用户关闭退出后再入场策略时保持现有行为

作为用户，我希望新能力默认不影响已有策略，以便安全加载并复现旧实验。

验收标准：

- Given 一个已有策略配置没有退出后再入场字段
- When 用户加载并运行该策略
- Then 系统将该能力解释为关闭
- And 不启动冷却、不暂存信号、不执行释放验证
- And 在相同数据、其余配置和计算版本下，交易序列及全部绩效指标与变更前完全一致
- Given 用户曾填写退出后再入场参数但随后关闭开关
- When 用户运行回测
- Then 已填写参数不参与计算
- And 页面摘要明确显示“退出后再入场限制：关闭”

### US9：冷却从退出实际成交日开始并按交易日计数

作为用户，我希望冷却期使用真实可成交日期和有效交易日计数，以免自然日或信号日造成偏差。

验收标准：

- Given 退出规则在交易日 T 收盘触发，退出在下一可用交易日 E 开盘成交
- And 冷却交易日数为 3
- When E 日成交完成
- Then E 为冷却第 0 日
- And E 之后第 1、2、3 个对齐有效交易日禁止进入 Risk Asset
- And 系统在第 3 个冷却交易日收盘后解除限制
- And 验证通过时最早在其下一可用交易日开盘再入场
- Given T 与 E 之间包含周末、休市日或被数据对齐丢弃的日期
- When 系统计数
- Then 这些日期均不计入冷却交易日
- Given 退出信号出现但退出交易未成交
- When 回测到达该日期
- Then 不启动冷却
- Given 入场规则恰好在第 3 个冷却交易日收盘触发
- When 系统先完成该日冷却计数并解除限制
- Then 该入场信号按冷却结束后的正常信号处理
- And 可以生成下一可用交易日开盘指令

### US10：用户可以忽略冷却期间的入场信号

作为用户，我希望选择忽略冷却期间出现的信号，以便冷却结束后只响应全新的入场机会。

验收标准：

- Given 再入场策略已启用
- And 冷却期间入场信号设为“忽略”
- When 冷却期间入场规则完整触发
- Then 不创建待处理信号或 T+1 指令
- And 策略事件记录一条 `Entry Ignored`
- When 冷却结束
- Then 系统不会因为此前被忽略的信号自动再入场
- And 必须等待冷却结束后出现新的正常入场信号

### US11：用户可以暂存冷却期间的最新入场信号

作为用户，我希望冷却期间保留最近一次入场信号，以便在限制解除后仍可利用尚未失效的机会。

验收标准：

- Given 再入场策略已启用
- And 信号处理为“暂存最新信号”
- And 保留期为 4 个交易日
- When 入场信号在交易日 S 收盘出现
- Then S 为有效期第 1 日
- And 信号在 S 及之后 3 个有效交易日内有效
- And 当前信号解释展示信号日期、最后有效日和剩余有效交易日数
- Given 暂存信号仍有效
- When 冷却期间出现一个更新的入场信号 S2
- Then S2 替换旧信号
- And 有效期从 S2 重新计算
- And 事件明细记录 `Signal Replaced`
- Given 同一交易日的入场规则组有多个子条件同时满足
- When 系统生成入场信号
- Then 只创建一个待处理信号

### US12：暂存信号会按明确边界到期

作为用户，我希望过期信号不会在很久以后意外成交。

验收标准：

- Given 一个暂存信号的最后有效日早于冷却结束日
- When 系统处理最后有效日收盘后的下一个有效交易日
- Then 信号状态变为 `Expired`
- And 冷却结束时不得基于该信号生成指令
- Given 信号最后有效日与最后一个冷却交易日是同一天
- When 系统处理该日收盘
- Then 先执行配置的释放验证
- And 验证通过时生成下一可用交易日开盘指令
- And 只有验证失败或未生成指令时才将信号结束为到期或拒绝状态

### US13：用户可以配置冷却结束后的释放验证

作为用户，我希望决定暂存信号在冷却结束后是否需要重新确认，以便在不同策略中控制信号陈旧风险。

验收标准：

- Given 用户选择“暂存信号仍有效即可”
- And 冷却结束时信号未到期
- When 系统执行释放判断
- Then 不额外要求入场规则在该日再次触发
- And 生成下一可用交易日开盘指令
- Given 用户选择“重新验证入场规则”
- When 冷却结束
- Then 系统以该日收盘数据重新计算完整入场规则组
- And 只有完整规则组满足时才生成指令
- Given 入场规则包含事件型交叉条件
- When 释放判断日没有再次发生该交叉
- Then 该事件型规则不满足
- Given 用户选择“通过释放验证规则组”
- When 冷却结束
- Then 页面与事件明细展示每条验证规则的值、结果和 AND/OR 汇总结果
- And 只有汇总结果满足时才生成指令
- Given 释放验证失败
- Then 信号标记为 `Rejected`
- And 不在后续日期自动重复验证同一个信号

### US14：再入场仍遵守统一的 T+1 成交口径

作为用户，我希望释放验证与真实成交分开显示，以免把当日收盘判断误解为已经持仓。

验收标准：

- Given 冷却在交易日 R 收盘结束
- And 暂存信号与释放验证均通过
- When 系统生成再入场指令
- Then R 只记录 `Release Passed` 和 `Order Scheduled`
- And R 日收盘后的实际持仓仍为 Fallback Asset 或 CASH
- And 下一可用交易日开盘才切换到 Risk Asset
- And 交易记录分别展示原始 Signal Date、释放判断日和 Execution Date
- Given R 的下一交易日缺少对齐后数据
- When 回测继续
- Then 指令顺延到下一可用交易日开盘执行
- And 交易与事件明细标注顺延

### US15：保存、加载和复制通用再入场配置

作为用户，我希望再入场策略随候选策略一起保存和复制，以便重复实验而不丢失规则。

验收标准：

- Given 用户已配置冷却、信号处理、保留期和释放验证规则
- When 用户保存实验、保存当前策略或收藏策略
- Then 保存对象包含所有已配置字段
- When 用户重新加载或从该候选复制新策略
- Then 所有字段、规则顺序、AND/OR 关系和自然语言摘要一致
- And 不恢复上次运行中的冷却进度、暂存信号或待执行指令

### US16：非法或边界参数被明确阻止

作为用户，我希望系统在运行前指出无法解释的参数，以免得到静默错误的回测结果。

验收标准：

- Given 再入场策略已启用
- When 冷却交易日数为空、非整数、小于 1 或大于 252
- Then 当前策略不能运行
- And 字段旁显示“请输入 1–252 的整数交易日”
- Given 信号处理为“暂存最新信号”
- When 保留交易日数为空、非整数、小于 1 或大于 252
- Then 当前策略不能运行
- And 字段旁显示“请输入 1–252 的整数交易日”
- Given 信号处理为“忽略”
- When 草稿中仍保留一个历史保留期数值
- Then 该数值不参与校验或回测
- Given 释放验证方式为“通过释放验证规则组”
- When 规则组为空、任一规则参数非法或引用的资产没有可用数据
- Then 当前策略不能运行
- And 错误定位到具体规则行
- Given 释放验证方式不是“通过释放验证规则组”
- When 草稿中仍保留历史验证规则
- Then 这些规则不参与校验或回测
- Given 冷却或保留期超过当前数据尾部
- When 回测到结束日期
- Then 回测可以完成
- And 当前状态显示剩余交易日未知或超出数据范围
- And 不伪造释放、到期或成交事件
- Given 任意数值字段包含小数、NaN、Infinity、科学计数法溢出或非数字文本
- When 用户尝试运行
- Then 当前策略被阻止并显示可操作的字段错误

## 12. 数据与指标口径

MVP 数据：

- 日线 OHLCV。
- 默认使用 Yahoo Finance chart API 或现有 Python 数据源。
- 前端必须显示数据源和最新完成日线日期。
- 回测必须按所有参与标的的共同交易日期对齐。
- 对齐方式为 inner join：只有所有必需标的都有数据的日期才进入回测。
- 页面必须显示原始数据区间和对齐后有效回测区间。

执行口径：

- T 日收盘确认信号。
- T+1 开盘成交。
- MVP 不模拟盘中触发。
- P0 执行方式固定，但必须在 UI 中显式展示。
- 冷却起点使用实际退出成交日，不使用退出信号日。
- 冷却与信号保留期均按完成 inner join 后的有效交易日计数，不按自然日计数。
- 冷却结束后的释放判断使用当日收盘可得数据；通过后仍在下一可用交易日开盘成交。
- 后续是否支持其他执行方式另行评估。

收益口径：

- 默认使用 close-to-close / open execution 的现有回测口径。
- 必须在结果中显示口径说明。

手续费与滑点：

- P0/P1 不支持手续费、滑点、spread 或税务设置。
- 页面可以在口径说明中写明“本回测未计入交易成本”。
- 成本模型不作为当前 Backtest Lab 范围。

## 13. 退出后再入场能力的分期边界

### P0

- 每个候选策略独立的启用/关闭开关，默认关闭并向后兼容旧配置。
- 统一的、以实际退出成交日为起点的固定交易日冷却期。
- 冷却期间信号处理支持 `忽略` 与 `暂存最新信号`。
- 一个待处理信号槽；新信号替换旧信号并重置有效期。
- 固定交易日保留期及确定性的到期边界。
- 三种释放验证方式：暂存信号有效即可、重新验证入场规则、通过独立释放验证规则组。
- 释放验证规则通过受支持的表单规则目录配置，支持 AND / OR。
- 冷却结束后的 T 日收盘判断 / T+1 下一可用交易日开盘成交。
- 配置区、自然语言摘要、当前状态解释、结果统计、交易记录关联和策略事件明细。
- 保存、加载、收藏、复制及旧配置向后兼容。

### P1

- `保留最早信号`、延长但不替换、有限队列等更多重复信号处理方式。
- 按退出原因或退出规则组配置不同冷却与释放策略。
- 在多策略排名中提供更完整的信号生命周期统计与筛选。
- 参数网格可批量搜索冷却期、保留期及释放验证参数。
- 从每周回测结果或每日指引配置导入、导出该策略。

### 非目标

- 不支持用户编写 Python、JavaScript、SQL、公式语言或任意表达式。
- 不提供脚本编辑器、插件式执行器或上传自定义策略代码。
- 不支持自然日、小时、分钟或自定义日历作为冷却单位；P0 只有对齐后的日线交易日。
- 不支持多个待处理信号的优先级、并行订单或多资产组合队列。
- 不改变资产角色模型；仍是 Signal / Risk / Fallback 之间的单持仓切换，不做权重组合或再平衡。
- 不改变 P0/P1 的成本模型、样本外验证、walk-forward、下单和高频回测边界。
- 不为任何特定 ticker、杠杆倍数、指标、冷却天数或信号保留天数提供隐藏的专用逻辑。

## 14. 风险与开放问题

风险：

- 参数太多会让页面像专业终端，反而降低使用效率。
- 如果数据源不稳定，用户可能误判策略表现。
- 回测结果容易造成过度自信，必须加入免责声明和风险提示。
- 把所有规则都做成 UI 会拖慢 MVP，应该先支持最常用的轮动策略。
- 冷却与信号暂存会增加状态解释复杂度；若 UI 没有明确区分“信号、释放、指令、成交”，用户可能误判当前持仓。
- 对事件型入场规则直接做“重新验证”可能非常严格，页面必须解释其与“释放验证规则组”的差异。

开放问题：

- P0 回测计算是在前端 JS 做，还是调用 Python backend/API？
- 保存策略 P0 是否只用 localStorage，还是直接写 Notion？
- 是否需要支持多个 benchmark？
- 是否需要把保存的策略接入每日 GitHub Action？
- 是否需要移动端适配，还是先做桌面研究工具？
- P0 释放验证规则目录中，除现有规则的当前状态形式外，首批还需要哪些通用状态条件？

已确认不进入当前范围：

- 手续费、滑点、spread 和税务建模。
- 资产可交易性、流动性、杠杆 ETF 衰减或相关性审查。
- 样本内/样本外、walk-forward 和参数稳定性验证。
- 策略生命周期管理。

PM 建议：

- P0 先使用现有 Python 回测逻辑作为权威计算来源。
- 前端页面只负责配置、展示和保存草稿。
- 如果短期没有后端，可以先让页面生成策略 JSON，并由本地/Action 脚本消费。
- 真正要做到 dashboard 内“一键运行”，建议后续增加轻量 backend API。

## 14. 成功指标

用户成功指标：

- 用户能在 1 分钟内配置并运行一个 QQQ/QLD 变体策略。
- 用户能在同一次实验中添加并比较最多 5 个策略候选。
- 用户能明确看出策略是否跑赢 QQQ buy & hold。
- 用户能理解每个策略最新信号为什么是持有、切换或退出。
- 用户能保存一个回测实验，并在下次打开时继续使用。

产品成功指标：

- 用户每周至少用该页面测试 1 个策略变体。
- 每周回测中排名靠前的策略能被加载到实验台继续分析。
- 每日指引使用的策略能从实验台保存或更新。

反指标：

- 页面不能让用户误以为策略收益可保证。
- 页面不能隐藏最大回撤和切换次数。
- 页面不能只展示收益最高的结果而不展示风险。

## 15. 非目标

- 不做真实交易。
- 不做投资建议。
- 不做全市场自动选股。
- 不做复杂脚本编辑器。
- 不做机构级风控系统。
- 不做社交分享或多人协作。
